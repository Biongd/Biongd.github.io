2. java语言概述

2.1 java技术

包括java语言和java平台

2.1.1 java语言

  1. 简单性：不使用指针并提供自动垃圾收集功能，程序员不必为内存管理担忧。

  2. 面向对象性：提供类，接口和继承等原语。

  3. 分布式(Distributed)：在基本java应用编程接口中有一个网络应用编程接口（java net），它提供了用于网络应用编程的类库，包括URL,URLConnection，Socket，ServeSocket等。

  4. 多线程(Multithreaded)：线程是一种特殊的对象，它的活动由一组方法来控制。线程的解释

  5. 动态性(Dynamic)：java类能够动态的载入到运行环境，也可以通过网络载入需要的类，有利于软件的升级

  6. 体系结构的中立性(Architecture Neutral)：程序在java平台被编译为体系结构中立的字节码格式，然后可以在装有java平台的任何系统中运行。这种途径适于异构的网络环境和软件的开发。
  7. 可移植性(Portable)：本身中立，规定了各基本数据类型的长度
  8. 高性能(High Performance)：与普通的解释型高级脚本语言相比，java的却是高性能的。
  9. 健壮性(Robust)：通过强类型检查、异常处理、垃圾回收等机制实现健壮性
  10. 安全性

2.2.2 java平台

由两部分组成：java虚拟机(JVM)和java应用程序接口（API）。

java平台将java程序和硬件平台隔离开，实现一次编写，到处执行的特点。

  1. JVM

    运行java代码的虚拟计算机，是一台通过软件模拟实现的真是计算机系统。有自己的硬件，如处理器，堆栈，寄存器等；还有相应的指令系统，为java字节码程序提供了独立的运行环境。

    执行字节码时，JVM需要把字节码解释成具体平台上的机器指令执行。所以只要安装对应平台上的JVM，java代码畅行无阻

  2. API

    java语言提供的组织成包接口的类和接口的集合（前接口非后接口），为开发者提供了各种有效的类和接口，并按照功能进行了分包存放。

3. 流（java.io）

3.1 概念(p/277)

  • 目的：越出JVM，或者说是内存，与外界进行数据交换。

    也就是将数据传输这一行为抽象为流这个概念，或者也可以具象的将流理解为数据的通道

  • 分类：

    • 数据流动方向划分：

      输入流和输出流，输入流只能从流里读不能写，输出流只能往流里写写不能读

      流一般成对出现（输入和输出），但也有特例：java.io.RandomAccessFile流类，既是输入流，又是输出流

    • 流的功能划分：

      节点流和功能流。节点流只能直接连接数据源，进行数据读写操作，功能流只能对已经存在的节点流进行连接和封装，通过封装增强流的读写功能和处理效率

    • 按流操作的数据单位划分

      字节流和字符流，其中以InputStrame和OutputStrame结尾的流为字节流，以Reader或Writer结尾的流为字符流。通过流结尾单词即可区分字节流和字符流类型

  • 体系结构

    4个顶层流类（抽象类）：InputStream和OutputStrame是字节流，Reader和Writer是字符流。均作为最顶层的父类，其中的抽象方法在子类中进行分化和实现。每个类中也定义了常用的基本方法。

3.2 流的使用

一、操作步骤

  1. 实例化流

    获得流对象，是进行流操作的前提。

  2. 包装流

    第一步实例化流对象用到的流类为节点流，构造流对象时和数据源建立连接。节点流处理效率较低，操作功能较弱，因此通过第一步获得的节点流对象使用功能流进行包装，以提高流的处理效率和增强流的功能。（p283，图10-8）

    我的理解：在硬盘和内存间加一个buffer功能流，避免总线占用，提高效率，再通过增加Data功能流，增加流对象能实现的作用。

    包装的方式就是给构造器传入节点流对象。

  3. 读写流

    即执行读或写操作

  4. 对流类型为输出流的流对象，清除缓存

    输出流类中的flush()方法可以强制输出buffer中缓存的内容，不过顶层输出类中该方法为抽象方法，只有实现该方法的流类实例对象才能调用该方法

  5. 关闭流

    释放系统资源，减轻内存压力

4. 异常（java.lang，java.io。。。）

4.1 概念

Error：JVM错误，内存资源耗尽等严重情况，一旦发生不可恢复，只能终止

Exception：由于编程错误或偶然因素引发的事件导致的一般性问题。

一、分类

java异常是异常类的对象，javaAPI中声明很多异常类，每个异常类都代表了一种运行错误，当java程序运行过程中发生一个可识别的运行错误时，系统都会产生一个相应的异常类对象，即产生一个异常

  1. 运行时异常（未检查型异常）

    这类异常不要求必须捕获，不进行异常处理，因此需要开发者小心谨慎，避免异常发生的情况

  2. 非运行时异常（检查型异常）

    强制要求开发者必须捕获和处理，否则编译报错

4.2 异常处理

一、异常的过程

抛出（throw）：生成异常对象并将其交给运行时系统的过程

捕获（catch）：从生成异常的方法开始回溯，直到找到异常处理的代码为止

处理：执行异常处理代码

处理机制：

  1. 就地处理，即捕获异常并处理异常；
  2. 向上抛出，即不进行捕获和处理异常，向上抛出给方法调用者

二、捕获并处理异常

try——catch——finally（一般用于资源的释放）

Exception类的getMessage（）获得异常的剪短信息；

getCause（）方法获取异常产生的原因

printStackTrace（）用来打印异常的发生点以及所有的调用点，即打印异常栈轨迹。该方法显示的错误信息比其他异常方法详细，指明了出错种类、具体位置和运行栈轨迹。

三、声明并抛出异常（消极的处理方式）

关键字：throws

5. 集合框架（java.util包）

5.1 概念（p/200）

一、集合：

存放一些互不相干的对象，但是实际开发中很少这么做。数组就是一种集合，不过数组中存放元素的类型必须相同，且可以存放基本数据类型和引用类型。集合中只能存放引用类型。

相同点：集合和数组一样，保存的都是一个对象的引用，而不是真正的对象数据，可以实现增删改查等操作

不同点：数组长度固定，集合中元素个数不确定，元素频繁增删，使用“下标索引”之外的方式检索元素等问题时，数组很难实现

二、集合框架

早在 Java 2 中之前，Java 就提供了特设类。比如：Dictionary, Vector, Stack, 和 Properties 这些类用来存储和操作对象组。虽然这些类都非常有用，但是它们缺少一个核心的，统一的主题。由于这个原因，使用 Vector 类的方式和使用 Properties 类的方式有很大不同。

集合框架被设计成要满足以下几个目标：

  • 该框架必须是高性能的。基本集合（动态数组，链表，树，哈希表）的实现也必须是高效的。
  • 该框架允许不同类型的集合，以类似的方式工作，具有高度的互操作性。
  • 对一个集合的扩展和适应必须是简单的。

即围绕一组统一的标准接口设计框架。你可以直接使用这些接口的标准实现，诸如： LinkedList, HashSet, 和 TreeSet 等,除此之外你也可以通过这些接口实现自己的集合。（对接口的意义理解一下升华了）

位置：集合相关的接口和类均处于java.util包下

5.2 集合框架结构

由于整个java集合框架的基础是集合接口，如Collection（单个元素）、Map（键值对）

5.2.1 Set（集）

模拟数学中集合的概念，是最简单的一种集合。

特征：

  • 不允许出现重复的元素
  • 无序，无下标

主要实现类：HashSet；子接口SortedSet的实现类：TreeSet

1.HashSet类

关键问题：

HashSet添加自定义学生类对象时，所添加的5个对象中有两个对象的传入参数完全一致，但是遍历集合时都可以输出，因为HashSet集合中不能存放重复的元素，这意味着这两个元素并不被认为是相同的？？？

理解：

因为HashSet的底层是采用哈希表实现，向HashSet添加对象元素时，会根据对象的哈希码计算对象的存储位置，对象的哈希码通过Object类提供的hashCode（）方法获得，hashCode（）方法返回的对象值根据对象的内存地址得到哈希码。这两个对象通过new构造的地址不一样，得到的哈希码就不一样！！如果用HashSet存放自定义类型的对象时，一定要重写hashcode（）和equals（）这两种方法，保证两个对象相同时，返回的哈希码相同，并且利用两个对象用equals（）比较时返回true。

2.TreeSet类

相比HashSet只是增加了排序功能的集合。会按照一定的排序规则，将集合汇总的对象序列时刻按照“升序”排列。所以必须注意的一点是：

  1. 向TreeSet添加的对象元素的类型必须实现处于java.lang包下的Comparable接口，否则程序运行时出现java.lang.ClassException异常。API中的String类，封装类都已实现该接口，这个接口只有一个抽象方法：public int compareTo(Object o)用来实现排序规则。

  2. 除了上面的类型，还可以使用定义比较器的方式实现定义排序规则。具体做法是：定义比较器类来实现java.util包下的Comparator接口，重写其中的compare（Object o1 , Object o2）方法来定义比较规则。然后在实例化TreeSet对象时调用TreeSet类的有参构造方法，向new的TreeSet对象传递一个比较器对象。

    例如“s.add（5）；”等价于“s.add(new Integer(5));”这中间发生了自动装箱，java API中的封装类实现了Comparable接口，即实现了比较规则。

综上：
TreeSet类不仅要满足HashSet的重写hashCode方法和equals方法，还有实现能够将集合内对象元素排序的接口。完整实例在书p212页。

例如“s.add（5）；”等价于“s.add(new Integer(5));”这中间发生了自动装箱，java API中的封装类实现了Comparable接口，即实现了比较规则。

6. 泛型

6.1 存在意义

书面：将原本确定不变的数据类型参数化。作为对原有Java类型体系的扩充，使用泛型可以提高Java应用程序的类型安全、可维护性、可靠性。

实例：集合框架中常遇到的数据造型问题

以Vector集合为例，从语法上讲，可以向其中添加任何类型的元素（基本类型数据会被自动封装为相应封装类的对象）。在JDK1.4以前，Vector添加元素的方法add()/addElement()的形参以及访问元素的方法get（）/elementAt（）的返回值均定义为Object类型，实际使用时，必须将从集合中读取的元素值再强制转换为所期望的类型，才能完全正常使用。

实际开发中，向某个集合加入的元素通常都是同一种类型的数据，而不会将多种不同性质和用途的信息(如学生信息，考试题目，考试成绩，图书信息)保存到同一个集合中，这样检索会很困难。这种情况下，每次取用集合元素时的例行操作只是增加开发者编码负担，无法杜绝取用元素时可能出现类型转换异常问题，因为集合本身不会对添加到其中的元素类型进行限制。例如，向保存人员信息（Person对象）的集合中误加入一个其他类型的元素时，编译阶段无法发现，程序运行时才会报错。

泛型的功能就是：允许编译器实施由开发者设定的附加约束类型，将类型检查从运行时挪到编译时运行，这样类型错误就可以在编译时暴露出来，而不是在运行时才发作。至于减少强制类型转换操作，只能算是泛型的一个附带好处。

6.3 进阶

6.3.1.类型参数

例如Vector<String>尖括号括起来的部分称为类型参数，而这种由类型参数修饰的类型被称为泛型类。为了便于理解，可以将泛型类理解为一种新的构造类型，也一定属于引用类型。

  • 除了异常类型，枚举或匿名内部类，任何类型都可以有类型参数。
  • 类型参数在定义时是形式类型参数，在使用时需指定类型，称为实际类型参数；和方法的形参，实参概念类似。
  • 如果使用泛型类时没有显式给出类型参数，则将默认作为Object类型处理。

实例的person类中，当你传入一个String数据类型时，就相当于将类中声明的变量类型secrecy定义为Sting。

与传参的区别：Java编码惯例，只使用少量的类型参数名称

  • K：键，比如映射的键
  • V：值，比如List和Set的内容
  • E：元素，比如Vector<E>
  • T：泛型，比如上面的Person类

6.3.2.类型通配符

不懂得作者们为什么都喜欢铺垫着循序渐进的讲解，可能是怕我们看完一点一知半解的就以为自己懂了吧，但是把实际内容藏的太深也会让人失去兴趣啊。。。

类型通配符，简而言之，就是一个”？“，这个问号配的谁？又为什么要用这个问号来作为通配符呢？

让我们从泛型类的角度出发：同一个泛型类搭配不同的类型参数复合而成的类型（如Person<String>/Person<Double>）属于同一个类，确是不同的类型，这是一种新情况。

泛型类可以和抽象类来类比：前者未确定的是其要处理的数据类型，后者未确定的则是其规划功能的具体实现。

注意：同一泛型类与不同的类型参数复合而成的类型间并不存在继承关系，即使是类型参数间存在继承关系也是如此。因为假如继承关系存在，父类和子类间无疑可以使用类型转换，即

泛型类可以“定型”为多种不同的具体类型，相互间又没有继承关系，这又会引发另一个问题，对于声明了形式类型参数的变量的方法无法通用。例如

7. GUI编程

GUI：Graphics User Interface（图形用户界面），区别于命令行窗口或控制台，与用户进行交互

7.1 基础

概述

三种GUI编程技术：Applet，AWT，Swing。使用Applet进行GUI编程，一般用于Web页面，目前很少使用，基本被Flash等技术取代。AWT不是甲骨文推荐的工具集，虽然在许多非桌面环境，如移动或嵌入式设备中有自己的优势，但是具有组件类型少，组件外观粗糙和以来本地操作系统等缺点。Swing是在AWT组件集的基础上进行的改进和升级，具有组件类型丰富，组件外形美观，与平台无关等优点。AWT：抽象窗口工具集，是JDK的一部分，也可以说是Java API的子集，包括组件、容器和布局管理器等GUI重要组件。AWT支持Java GUI事件处理机制。

1.常用的Java API相关组件包：

java.awt包，使用不多，多被Swing取代；java.awt.event包，事件处理包，实现程序界面与用户的交互。javax.swing包，是图形界面组件升级包，是对java.awt.*包中的GUI组件的扩展和升级，提供种类丰富、外形美观的GUI组件，在目前的Java GUI开发中基本取代了大多数AWT组件，应用广泛。

2.组件

图形用户界面的基本组成元素，除了容器类组件外，其他组件不能单独显示，必须放在一个容器内才能显示。java.awt.*包定义了很多组件类：Frame、Button、Menu、Lable、TextField等。java.awt.*包中有一个抽象组件类java.awt.Component，是AWT组件中除了Menu类之外所有组件类的功能父类。

3.容器

容器（Container）本身也是一个组件，具有组件所具有的所有性质，只是容器组件可以容纳其他组件或容器，所以Container是Component的子类。所有容器都具有add（）方法，用于将其他组件添加到容器中。

容器又分为可以独立存在的顶层容器和必须依附于其他容器存在的非顶层容器。java.awt.Window为顶层容器类，是可以自由停泊的。如Frame就是一个顶层容器。java.awt.Panel为非顶层容器，可以作为容器容纳其他组件，但不能独立存在。

java.awt.Frame是最常见的顶层容器，其层次类型结构为

    B --> C[java.awt.Container]

4.布局管理器

控制组件的排列风格，包括组件位置和大小。其存在也是为了使GUI具有良好的平台无关性。java.awt.Window及其子类（Frame，Dialog）的默认布局管理器为BorderLayout，Panel及其子类（Applet）的默认布局管理器为FlowLayout，并且一个容器只能有一种排列风格，即一个布局管理器。

常见的4中布局管理器：边界布局管理器（BorderLayout）、流式布局管理器（FlowLayout）、网格布局管理器（GridLayout）与卡片布局管理器（CardLayout）。

5.容器嵌套

因为一个容器只能使用一种布局管理方式，所以对于复杂的图形界面，往往需要进行多容器的嵌套。

7.2 GUI事件处理

交互的实现才是建立图形界面的根本目的

1.事件对象

事件被触发，组件就会产生对应的事件对象（有java语言直接提供，开发者很少构建事件对象类）。例如对按钮进行单击操作，按钮组件就会产生ActionEvent事件对象。

2.事件源

能够接收用户操作并产生事件对象的GUI组件

3.事件监听器（p/162）

对事件进行处理的对象，该对象定义了能够接受，解析和处理事件的方法，方法实现与用户的交互。每一种事件都对应专门的监听器中的某一方法，监听器中的方法负责接受和处理这种事件。

总结：GUI的开发分为以下4个步骤

  • 确定GUI界面的容器；

  • 设置容器的布局管理器

  • 向容器中添加需要的组件
  • 为组件添加事件处理

4.GUI事件类型

java.lang.Object—>java.util.EventObject—>java.awt.AwtEvent

5.适配器类

是对应大部分事件监听器接口定义的对应的实现类，只是所有方法都是空实现，即什么都不做。这样就可以通过继承适配器类，重写响应的事件监听代码，不需要像实现监听器接口一样写大量的空实现。

又由于java不支持多继承，所以当有事件源需要绑定多个事件监听器时，就必然需要事件监听器接口了。

8. 数组

首先回忆一下集合和数组的区别：

  • 数组中存放的元素类型必须相同；集合中可以存放一些互不相干的对象，但是开发中一般不会这样做（不然为啥有泛型）
  • 数组中可以存放基本数据类型和引用类型；集合中只能存放引用类型
  • 最关键的一点，数组是定长的！集合可以调用方法增删元素！

本质上作为一对元素的存储体，两者都只是存储了对象的引用，而非对象的实际数据。
